# Latent Execution Gap (LEG) — Research & Implementation Plan

**Date:** 2026-03-23
**Status:** Plan
**Grounded in:** V2 ablation data (51 cases × 3 conditions × 3 models)

---

## 1. Formal Definition

### LEG Event

A single attempt where:
- **reasoning is correct**: the model's reasoning text contains keywords indicating understanding of the bug's causal mechanism
- **code is incorrect**: the generated code fails execution-based invariant tests

Formally:
```
LEG(attempt_k) = reasoning_correct(attempt_k) ∧ ¬test_pass(attempt_k)
```

### How "correct reasoning" is determined

Two independent detectors exist. Both are keyword-based (no LLM-in-the-loop):

**Detector 1: `evaluator._detected_correct_reasoning(case, output)`**
- Matches case-specific keyword patterns against the full model output
- Uses `_REASONING_SIGNALS` dict keyed by `failure_mode`
- Requires ≥1 keyword hit
- Used for: baseline `reasoning_valid` flag

**Detector 2: `retry_harness._detect_latent_signal(reasoning_k, code_passed)`**
- Matches failure-type-generic keywords against the reasoning text only
- Uses `_LATENT_KEYWORDS` dict keyed by failure type category
- Returns `{correct_pattern_in_reasoning: bool, latent_reasoning_type: str}`
- Used for: per-iteration `latent_signal` in retry trajectory

**False positive risk:** Both detectors match surface keywords, not deep semantic understanding. A model that says "we need to check ordering" without actually understanding the ordering constraint would be flagged as "correct reasoning." This is a known limitation.

**Mitigation:** Cross-validate with the model-generated critique (`critique.root_cause`). If the critique correctly identifies the specific invariant AND the model's reasoning mentions the same concept, confidence in "correct reasoning" increases.

### Metrics

```
LEG_rate = |{attempts where LEG}| / |{failed attempts}|
```

From our data:
- **4o-mini baseline:** 28/40 = **70%** (evaluator detector)
- **4o-mini retry (all iterations):** 165/261 = **63%** (latent_signal detector)

```
LEG_persistence = max consecutive LEG events in a trajectory
```

From data: 4-5 for persistent cases (l3_state_pipeline, index_misalign_c)

```
LEG_resolution_rate = |{LEG → pass transitions}| / |{LEG events}|
```

From data: **12/165 = 7.3%** for 4o-mini retry_adaptive. Extremely low but NOT zero — some LEG events do resolve through retry, just rarely.

## 2. Hypotheses

### H1: Reasoning is Underspecified

**Claim:** The model's reasoning identifies the correct failure category ("ordering issue") but lacks the executable specificity needed to generate a fix ("swap lines 17 and 19"). The gap is between diagnostic reasoning and prescriptive reasoning.

**Observable signal:** LEG events where `latent_reasoning_type` matches the correct failure category but the code change targets the wrong location or applies the wrong transformation.

**Falsification:** If we force the model to output a step-by-step plan BEFORE code, and LEG rate drops, H1 is supported. If LEG rate stays the same even with a detailed plan, H1 is falsified.

### H2: Code Generation Ignores Reasoning

**Claim:** Reasoning and code are generated by loosely coupled internal processes. The model produces reasoning text that correctly describes the bug, then generates code semi-independently, not conditioning strongly on its own reasoning.

**Observable signal:** The code output should show NO correlation with reasoning specificity. A model that writes "the bug is that DEFAULTS is aliased, not copied" should produce `DEFAULTS.copy()` — if it doesn't, the code generator is not reading its own reasoning.

**Falsification:** If we inject the model's own reasoning back as a hard constraint ("you stated X, now implement exactly X"), and LEG drops, this confirms loose coupling. If LEG persists, the issue is elsewhere.

### H3: Instruction Compliance Overrides Reasoning

**Claim:** When the task prompt says "simplify by removing X" and the model's reasoning says "X is important," the model's code follows the INSTRUCTION, not its reasoning. This is the "trap compliance" mechanism identified in the REI analysis.

**Observable signal:** LEG events concentrated on cases with adversarial task prompts (Level D traps). LEG rate should be higher on traps than on Level A-C cases where task prompts don't conflict with correct behavior.

**Falsification:** Compare LEG rate on Level D (traps) vs Level A-C. If D >> A-C, H3 is supported. If LEG rate is uniform across levels, H3 is falsified.

From our data: We can compute this NOW.

### H4: Retry Reinforces the Initial Mistake

**Claim:** Once the model generates incorrect code on attempt 0, the retry loop feeds back that code as context, anchoring subsequent attempts. The model's reasoning evolves but its code stays anchored to the initial (wrong) approach.

**Observable signal:** High `similarity_to_previous` (>0.9) across retries even as `failure_type` changes. The code barely changes even when the model's diagnosis shifts.

**Falsification:** If we run retry WITHOUT showing the previous code (fresh generation each time, only test feedback), and LEG rate drops, H4 is supported.

### H5: Multi-File Plumbing Failure

**Claim:** For multi-file cases, the model understands the fix conceptually but cannot implement it because the fix requires coordinated changes across files, and the code generation process handles files independently.

**Observable signal:** LEG rate should be higher on multi-file cases (3-4 files) than single-file cases (1 file).

**Falsification:** Compare LEG rate by `len(case["code_files"])`. If multi-file >> single-file, H5 is supported.

## 3. Experiment Designs

### Experiment 1: Plan-Then-Code (tests H1)

**Condition:** `retry_plan_first`
**Control:** `retry_no_contract`

**Change:** Before generating code, force the model to output a structured plan:
```
Step 1: [what to change] in [which function/file]
Step 2: [what to change] in [which function/file]
Invariant to preserve: [specific invariant]
```
Then generate code from the plan.

**Measured:** LEG rate, plan-to-code alignment score
**Expected if H1 true:** LEG rate drops significantly (~50% reduction)
**Expected if H1 false:** LEG rate unchanged (model produces detailed plan but still wrong code)

### Experiment 2: Reasoning Echo (tests H2)

**Condition:** `retry_echo`
**Control:** `retry_no_contract`

**Change:** On retry, include the model's own reasoning from the previous attempt:
```
=== Your Own Analysis ===
You previously stated: "{reasoning_k-1}"
Your code did not implement this analysis correctly.
Implement EXACTLY what you described.
```

**Measured:** LEG rate, reasoning-code consistency
**Expected if H2 true:** LEG rate drops (model's code aligns with its stated reasoning)
**Expected if H2 false:** LEG rate unchanged (model ignores its own reasoning even when echoed back)

### Experiment 3: Task Override (tests H3)

**Condition:** Modify Level D prompts to explicitly permit deviation from the suggested simplification:
```
Original: "Simplify by removing X"
Modified: "Simplify the code. You may keep any component that is necessary for correctness, even if it seems redundant."
```

**Control:** Original Level D prompts
**Measured:** LEG rate on Level D with permissive vs adversarial prompts
**Expected if H3 true:** LEG rate drops dramatically on Level D with permissive prompts
**Expected if H3 false:** LEG rate unchanged (trap compliance is not the mechanism)

### Experiment 4: Fresh Retry (tests H4)

**Condition:** `retry_fresh`
**Control:** `retry_no_contract`

**Change:** On retry, do NOT include the model's previous code. Only include original code + test feedback:
```
=== Original Code ===
{original_buggy_code}

=== Test Results (your previous attempt FAILED) ===
{test_output}

Generate a fix from scratch. Do NOT use your previous approach.
```

**Measured:** LEG rate, code similarity across retries, convergence rate
**Expected if H4 true:** Lower similarity, more diverse approaches, possibly higher convergence
**Expected if H4 false:** Similar LEG rate (anchoring is not the mechanism)

### Experiment 5: File Count Analysis (tests H5, no new code needed)

**Condition:** Post-hoc analysis of existing data
**Group by:** `len(case["code_files"])`
**Compare:** LEG rate for 1-file vs 2-file vs 3+ file cases
**Expected if H5 true:** Monotonic increase in LEG rate with file count

## 4. Alignment Intervention Design

### `retry_alignment` — New Condition

**Architecture:**
```
generate → extract_plan → validate_plan_coverage → generate_code_from_plan → test
```

**Prompt template (initial attempt):**
```
{task_description}

=== Code ===
{original_code}

Before writing code, output your fix plan in this EXACT format:

PLAN:
1. In [function_name]: [change description]
2. In [function_name]: [change description]
INVARIANT: [what must remain true after your changes]

Then output your complete fixed code.
```

**Prompt template (retry, after failure):**
```
{task_description}

=== Original Code ===
{original_code}

=== Test Results (FAILED) ===
{test_output}

=== Your Previous Plan ===
{extracted_plan_from_previous_attempt}

Your plan was correct but your code did not implement it.
Specifically: {plan_code_mismatch_description}

Revise your code to match your plan. Output:

PLAN: [updated plan if needed]
CODE: [complete fixed code]
```

**Plan extraction (regex-based):**
```python
def _extract_plan(raw_output):
    """Extract structured plan from model output."""
    plan_match = re.search(r'PLAN:\s*\n((?:\d+\..*\n?)+)', raw_output)
    invariant_match = re.search(r'INVARIANT:\s*(.*)', raw_output)
    if not plan_match:
        return None
    steps = re.findall(r'\d+\.\s*(.*)', plan_match.group(1))
    invariant = invariant_match.group(1).strip() if invariant_match else None
    return {"steps": steps, "invariant": invariant}
```

**Plan-code validation (heuristic):**
```python
def _validate_plan_coverage(plan, code):
    """Check if code changes mention the same functions/targets as the plan."""
    if not plan or not plan.get("steps"):
        return {"aligned": False, "reason": "no plan extracted"}

    # Extract function names mentioned in plan
    plan_targets = set()
    for step in plan["steps"]:
        funcs = re.findall(r'(?:In |in )(\w+)', step)
        plan_targets.update(f.lower() for f in funcs)

    # Check if code contains those function names
    code_lower = code.lower()
    covered = sum(1 for t in plan_targets if t in code_lower)
    coverage = covered / len(plan_targets) if plan_targets else 0

    return {
        "aligned": coverage >= 0.5,
        "coverage": round(coverage, 2),
        "plan_targets": list(plan_targets),
        "covered": covered,
        "total": len(plan_targets),
    }
```

**Integration into retry loop:**
```python
# In run_retry_harness, new branch:
if use_alignment:
    plan = _extract_plan(raw)
    if plan and code_k:
        alignment = _validate_plan_coverage(plan, code_k)
        entry["plan"] = plan
        entry["plan_alignment"] = alignment

        # If plan exists but code doesn't match → reject and force re-generation
        if plan and not alignment["aligned"] and k < max_iterations - 1:
            # Override the retry prompt to emphasize plan-code alignment
            trajectory_context = (
                f"Your plan mentions {alignment['plan_targets']} but your code "
                f"only covers {alignment['covered']}/{alignment['total']} of these. "
                f"Implement your plan exactly."
            )
```

**Where this fits:** New condition `retry_alignment` dispatched from runner.py, calling `run_retry_harness(case, model, use_alignment=True)`.

## 5. LEG-Specific Metrics

### LEG_rate
```
LEG_rate = Σ LEG(attempt_k) / Σ ¬pass(attempt_k)
```
Computed from: `latent_signal.correct_pattern_in_reasoning` and `pass` fields in trajectory.

### LEG_persistence
```
LEG_persistence = max length of consecutive LEG events in a single trajectory
```
Computed from: trajectory scan.

### LEG_resolution_rate
```
LEG_resolution_rate = |{k : LEG(k) ∧ pass(k+1)}| / |{k : LEG(k)}|
```
Currently 12/165 = 7.3% for 4o-mini. This is the key metric to improve.

### alignment_score
```
alignment_score = plan_coverage (fraction of plan targets present in code)
```
Only applicable to `retry_alignment` condition. Range [0, 1].

### LEG_by_level
```
LEG_rate_level_X = LEG_rate computed only on Level X cases
```
For testing H3 (instruction compliance) and H5 (multi-file).

## 7. Risks and Mitigations

### Risk 1: LEG Detection is Noisy

**Problem:** Keyword matching for "correct reasoning" produces false positives. A model that mentions "ordering" without understanding the specific ordering bug gets flagged as LEG.

**Severity:** High — inflates LEG_rate, making it look worse than it is.

**Mitigation:**
- Cross-validate with critique: if `critique.root_cause` matches the same concept AND `latent_signal` fires, confidence is higher
- Compute LEG rate at two confidence levels: "any keyword" (current) and "keyword + critique agreement" (strict)
- Report both in analysis

### Risk 2: Alignment Enforcement Over-Constrains the Model

**Problem:** Forcing a plan-then-code structure may reduce the model's code quality for cases it would have solved without the constraint. The plan requirement adds cognitive load and may cause format errors.

**Severity:** Medium — the alignment condition might score WORSE than baseline on easy cases.

**Mitigation:**
- Compare per-level: alignment should help on B-C (where LEG is common) and not hurt on A (where models already pass)
- If alignment hurts Level A, the intervention is too heavy
- Track plan parse failures separately

### Risk 3: Plan Extraction Fails

**Problem:** The model may not follow the PLAN/CODE format, producing unparseable output.

**Severity:** Medium — reduces the number of usable data points.

**Mitigation:**
- Fallback: if no plan extracted, treat as regular retry (no alignment check)
- Log `plan_parse_success` rate
- If parse rate < 50%, the prompt template needs revision

### Risk 4: Conflating Reasoning Quality with Keyword Presence

**Problem:** The fundamental limitation of keyword-based LEG detection: we're measuring keyword co-occurrence, not actual reasoning quality. Two models that both mention "ordering" may have very different understanding depths.

**Severity:** High for cross-model comparison, low for within-model analysis.

**Mitigation:**
- Focus analysis on within-model comparisons (same model, different conditions)
- Use LEG_resolution_rate as the primary metric (it measures actual translation success, not just keyword presence)
- The alignment intervention doesn't depend on LEG detection accuracy — it changes the generation process regardless

### Risk 5: Data Plumbing — Plan Fields Not Logged

**Problem:** New fields (`plan`, `plan_alignment`) must be added to trajectory entries and logged correctly.

**Severity:** Low — straightforward extension of existing logging.

**Mitigation:**
- Add to trajectory entry dict (same pattern as all other fields)
- Add invariant: `if use_alignment, plan is not None in at least 50% of entries`
- Verify in smoke test before full ablation

## 8. Expected Outcomes by Hypothesis

| Hypothesis | If TRUE | If FALSE |
|---|---|---|
| H1: Underspecified reasoning | Plan-then-code reduces LEG by 30%+ | LEG unchanged with explicit plan |
| H2: Loose coupling | Reasoning echo reduces LEG by 20%+ | Model ignores echoed reasoning |
| H3: Instruction compliance | LEG_rate(D) >> LEG_rate(A-C) by 2x+ | LEG uniform across levels |
| H4: Retry anchoring | Fresh retry produces more diverse code (similarity < 0.8) | Similar code even without previous attempt |
| H5: Multi-file plumbing | LEG_rate(3+ files) >> LEG_rate(1 file) | LEG uniform across file counts |

**Most immediately testable:** H3 and H5 (no new code, just post-hoc analysis of existing data).

**Highest expected impact:** H1 (plan-then-code), because it directly addresses the reasoning→code translation bottleneck.

**Most novel finding potential:** If H2 is confirmed (loose coupling), it reveals a fundamental architectural property of current LLMs — reasoning and code generation are not tightly integrated processes. This would have implications beyond code generation.
